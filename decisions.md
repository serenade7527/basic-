# 데이터 정제 및 처리 기준 (keywords_input.csv, 25건)

## 1. detail 결측 (2건: id 012, 020)
- 처리 기준: `detail`이 비어 있어도 `type` · `keyword` · `brand_name`만으로 채널별 초안 생성 가능하다고 판단.
- 실제 처리: id 012(강사 만족, 수업 준비 / 영어마을 어학원), id 020(리뷰 작성 이벤트, 기프티콘 / 똑똑한 토론학원)은
  `type` 전략(후기 → 인터뷰형 / 이벤트 → 참여 방법+혜택)과 `keyword`·`brand_name`을 기반으로 3채널 초안을 정상 생성함.
- 케이스 헤더에 "(detail 결측 → type·keyword·brand_name 기반으로 생성)" 표기로 결측 처리 사실을 명시.

## 2. 완전 중복 행 (1건: id 025 = id 001)
- 확인: id 025는 `type`·`keyword`·`tone_hint`·`brand_name`·`detail` 전 컬럼이 id 001과 동일.
- 처리 기준: 동일한 입력으로 다시 콘텐츠를 생성하면 실질적으로 같은 글이 되어 리소스 낭비이자 채널 내 자기복제 콘텐츠가 되므로,
  025는 별도 초안을 생성하지 않고 001 초안을 재사용하도록 처리. 3개 초안 파일 상단에 처리 기준을 고지.

## 3. tone_hint 표기 흔들림 (1건: id 017 "긴박")
- `rules/tone_normalization.json`의 별칭 테이블에 따라 "긴박" → "긴급한"으로 정규화.
- 다른 긴급 케이스(008, 014)와 동일하게 "긴급한" 톤 전략(기간 한정·마감 임박 강조, D-day 카운트 등)을 적용해 톤 표기 차이로 인한
  전략 불일치가 발생하지 않도록 처리.

## 4. 품질 점검 결과 (scripts/build_drafts.py 실행 로그)
- 총 24개 케이스 생성 (025는 001 재사용 처리로 제외)
- 블로그 24건 모두 500자 이상 통과 / 인스타 24건 모두 150자 이내 + 해시태그 5개 통과 / 카카오 24건 모두 180자 이내 + 이모지 1~2개 통과
- 채널 간 복붙 없음: 동일 케이스라도 블로그(정보·신뢰형 서술) / 인스타(공감 한 줄 + 해시태그) / 카카오(이모지+행동유도 단문)로
  구조와 문장을 다르게 작성함
